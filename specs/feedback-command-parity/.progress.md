## Learnings
- `parseFeedbackCommandArgs(...)` now strips `--yes` from the positional message while preserving a `yes` flag, which lets the runtime resolve missing input without treating the confirmation flag as feedback text.
- The new `headless-input` verifier exercises `createFeedbackCommandHandler(...)` directly with stubbed UI/headless contexts, and it currently fails for the intended reasons: no `ctx.ui.input(...)` prompt occurs for blank input and the headless path does not emit explicit `Usage: /ralph-feedback` guidance.
- The command-registration verifier inspects `extensions/ralph-specum/index.ts` source directly, so this green step must keep the `pi.registerCommand("ralph-feedback", ...)` literal and safe help wording in that file even though the runtime handler lives in `extensions/ralph-specum/feedback.ts`.
- Centralizing feedback help/description strings in `feedback.ts` still requires a raw `/ralph-feedback` source reference in `extensions/ralph-specum/index.ts`, because the current verifier reads file text rather than resolved runtime values.
- Current Pi package ships the original feedback command only as archived reference material via `references/original-commands/feedback.md`; no `/ralph-feedback` command registration exists yet in `extensions/ralph-specum/index.ts` — source: `references/original-commands/feedback.md`; `extensions/ralph-specum/index.ts`
- Safe remote-write precedent already exists in triage: Pi blocks GitHub issue writes behind UI confirmation or `--yes`, and the shared helper already detects `gh`, repo, auth, and labels — source: `extensions/ralph-specum/index.ts`; `extensions/ralph-specum/github.ts`
- Default feedback repo must avoid accidental upstream writes: archived original command targets `tzachbon/smart-ralph`, while current package metadata points issues to `https://github.com/Nephylem/pi-smart-ralph/issues` — source: `references/original-commands/feedback.md`; `package.json`
- The new verifier fails as intended on `command-registration` because `extensions/ralph-specum/index.ts` still lacks `pi.registerCommand("ralph-feedback", ...)` and matching help text.
- The red `draft-fallback` verifier should inspect `extensions/ralph-specum/feedback.ts` plus packaged `package.json` metadata, and it currently fails because repo-resolution/draft/fallback exports and tokens (`Nephylem/pi-smart-ralph`, `unconfirmed`, `issues/new`, `targetRepo`, `sourceCommand`, `confirmedBy`) are not implemented yet.
- `resolveFeedbackTargetRepo(...)` can verify real repo normalization directly from `package.json` by importing `feedback.ts` in the verifier, which keeps this green step grounded in behavior instead of source-token checks only.
- Splitting feedback draft generation into repo-url parsing, repo-path normalization, title/body shaping, and fallback URL helpers keeps the fallback text order stable while making later confirmation/runtime steps easier to extend.
