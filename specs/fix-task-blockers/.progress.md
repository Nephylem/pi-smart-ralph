# Progress: fix-task-blockers

## Original Goal

Patch Smart Ralph / ralph-specum to stop false blockers in split-repo and nested-repo workspaces. Problem - Ralph currently assumes one task can always make one commit containing: - task implementation/test files - <basePath>/tasks.md - <basePath>/.progress.md - That fails when specs are outside git or code lives in multiple nested repos. - It also misclassifies some successful [RED] tasks as failures when the verify step intentionally expects failing tests. Required fix 1. Add git-topology detection For each task, detect repo roots for: - task-listed files - <basePath>/tasks.md - progress file Classify: - single_repo - multi_repo - repo_plus_nonrepo - no_repo 2. Make commit handling topology-aware - Honor `Commit: None` strictly: no commit required. - In `single_repo`, keep current behavior. - In `multi_repo`, `repo_plus_nonrepo`, or `no_repo`, do not hard-block only because one combined commit is impossible. - Allow success with: - `commit: none` + reason like `split_repo_workspace` - or multi-repo commit reporting if implemented 3. Fix RED-task completion validation - For `[RED]` tasks, if verify output explicitly proves expected failure success (e.g. `RED_PASS`), treat the task as successful. - Do not report raw failing-test output as the root blocker when the real issue is commit topology. 4. Improve blocker reason priority Prefer: 1. explicit workspace/commit-topology blocker 2. explicit TASK_MODIFICATION_REQUEST reason 3. explicit verification failure 4. missing completion signal fallback 5. Update executor + planner prompts - Executor must preflight repo topology before commit. - Executor must understand split-repo/spec-outside-repo cases. - Planner should avoid impossible commit contracts and default to `Commit: None` when spec files and task files cannot be committed together. Acceptance criteria - Single-repo tasks behave as before. - `[RED]` task with expected failing tests + `RED_PASS` completes successfully. - Split-repo task with `Commit: None` completes without blocking. - Split-repo task with a commit message does not hard-block solely due to impossible combined commit scope. - Blocker messages report the true cause. Patch likely areas - ralph-specum validation/blocker logic - commit handling helpers - Ralph spec executor prompt - Ralph task planner prompt - tests for repo topology + RED_PASS behavior
